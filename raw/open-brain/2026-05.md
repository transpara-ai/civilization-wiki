Adversarial review of transpara-ai/eventgraph PR #40 (E2 capability evolution MVP, SHA 14e90c3bf18a26008d7f0eed03123b55b46fe0f6, branch codex/e2-capability-evolution). Posted as a `COMMENTED` PR review (not request-changes): https://github.com/transpara-ai/eventgraph/pull/40#pullrequestreview-4305624880.

Architectural pattern at the root of most blockers: "helper-only gates + reverse-lookup evidence + open AppendEdge surface". The gates live inside `PromoteCapabilityVersion` / `ActivateCapabilityVersion` helpers, but the underlying `Store.AppendRecord` and `Store.AppendEdge` (`go/pkg/darkfactory/v39/store.go:40,81`) have no type-aware predicates â€” so direct AppendRecord/AppendEdge bypasses every gate. And the evidence chain on `CapabilityVersion` has no explicit foreign keys to its CandidateVariant/OptimizationRun/EvalDataset/BenchmarkResult/HumanReview; the gates resolve them by `ByType` iteration and first-match. Once one clean evidence chain exists for an artifact, every subsequent version inherits it.

The six blockers, in severity order:
1. `ActivateCapabilityVersion` (`capability_evolution.go:111-154`) is independent of `PromoteCapabilityVersion` â€” no PROMOTED_TO edge check, no re-eval of promotion evidence. Direct AppendRecord of a CapabilityVersion with Status="approved" + a non-nil RollbackTo + an FRV that lists it is sufficient to activate. Defeats human-review/benchmark-regression/optimizer-actor constraints.
2. `findCandidateVariantForArtifact` (cap_evo.go:282) and `findBenchmarkResultForCandidate` (cap_evo.go:292) use first-match ByType iteration. Evidence reuse across versions on the same artifact.
3. `findApprovedHumanReview` (cap_evo.go:302-321) third fallback accepts any approved HumanReview whose `SourceRefs` contains the artifact's CommonNode.ID, ArtifactID, or candidate ID â€” string-contains heuristic, not a typed reference. One fabricated approved review satisfies the gate forever.
4. `AppendEdge` (store.go:81) has no edge-type allowlist. Governance edges USED_CAPABILITY, REVIEWED_BY, PROMOTED_TO, ACTIVATED_BY, ROLLED_BACK_TO can be synthesized directly. RecordCapabilityUsage's UsageLoggingRequired check is bypassable.
5. `capabilitySourceRefs` (cap_evo.go:323-339) is opt-in via 10 case-sensitive literal prefixes. Source refs outside the list (case mismatch, bare ID, alternate separator, HTTPS URL, etc.) silently set `path.Completed=true` with no Missing entries. Hard constraint 8 becomes opt-in.
6. `findCapabilityArtifact` (cap_evo.go:256) falls back to matching on `Name` â€” a free-form display string without uniqueness enforcement in `Validate`.

Serious issues: canary 100% / empty AllowedProjects not blocked; RecordRollbackRecord doesn't transition status (rolled-back versions can be reactivated); FRV.CapabilityVersionRefs existence-only validated (no status check); optimizer-actor block is bare string equality; TOCTOU on concurrent Promote; ActivationPolicy.Scope="global" accepted by Validate but rejected at runtime (partially-valid anti-pattern that CLAUDE.md forbids); test suite doesn't cover bypass conditions and `RecordRollbackRecord` is fully untested.

Moderate: `UsageLoggingRequired != true` in Validate is a silent breaking change for Stage-1 records persisted with default `false`; CapabilityUsageEvidencePath double-appends orderPath.Missing; three unused mustGet helpers in query.go; EvolutionOrderID-only Tasks (validate.go:202-204) are invisible to certification paths.

Recommended top fixes: (1) add explicit foreign keys on CapabilityVersion to its candidate/run/dataset/benchmark/review; (2) make Activate require a PROMOTED_TO edge or re-run evaluateCapabilityPromotion; (3) gate governance edges in AppendEdge or remove the public surface for them; (4) invert capability-usage detection â€” enumerate required CapabilityArtifacts from the BOM and require USED_CAPABILITY per artifact, instead of parsing Task.SourceRefs prefixes; (5) enforce a state machine on CapabilityVersion in the store (AppendRecord allows only "approved" initial); (6) tighten ActivationPolicy.Validate to forbid "global" and require non-empty scope-restriction sets; (7) status-validate FRV.CapabilityVersionRefs; (8) drop Name fallback from artifact lookups; (9) add the bypass-shape tests listed in S7.

Five parallel reviewers (general-purpose Sonnet) converged independently on blockers B1-B5 and serious S7. PR #38's Medium-2 (undocumented source-ref prefix contract) recurs at scale in this PR â€” the contract is now 10 prefixes. PR #38's Medium-1 (invariants must hold at AppendRecord, not just helpers) is the same architectural shape as B1 and B4.
